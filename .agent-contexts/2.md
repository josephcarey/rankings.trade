# Card #2: Add D1 binding and forward-only migration runner — Implementation Summary

## What was built

A fully functional D1 database binding with a forward-only, run-once migration system for the rankings.trade Worker:

1. **D1 Binding** (`wrangler.toml`): Declared D1 database binding `DB` pointing to local development database `rankings-trade-dev`
2. **Migration Runner** (`src/lib/db/migrate.ts`): Core async runner that:
   - Validates that the first migration creates the `_migrations` bookkeeping table
   - Tracks applied migrations to ensure idempotency
   - Applies pending migrations in lexical order
   - Stops with clear errors on SQL violations or semantic issues
3. **Migration Loader** (`src/lib/db/loader.ts`): File-system loader that reads `.sql` files from `migrations/` directory, sorted numerically
4. **CLI Script** (`scripts/db-migrate.ts`): Functional `bun run db:migrate` entry point that:
   - Loads migrations from disk
   - Accepts D1 binding from worker context
   - Outputs clear logging of applied migrations
5. **Initial Migration** (`migrations/0001_init.sql`): Creates the `_migrations` bookkeeping table with name, applied_at columns
6. **Test Suite** (`src/lib/db/migrate.test.ts`): 17 comprehensive tests using real SQLite (sql.js) that validate:
   - Table creation and schema enforcement
   - Idempotency on repeated runs
   - SQL semantics (e.g., detecting duplicate CREATE TABLE attempts)
   - Multiple statement handling
   - Partial migration runs (new migrations added to existing database)

## Key decisions made

1. **No pre-creation in runner**: Removed the `CREATE TABLE IF NOT EXISTS _migrations` from `runMigrations()`. Instead, the runner validates that the first migration (with "init" in the name) creates this table. This enforces that migration files are the source of truth for schema.

2. **Real SQLite in tests, not mocks**: Replaced the hand-rolled `MockD1Database` with real SQLite via `sql.js` (in-memory implementation). The mock was silently ignoring SQL errors (e.g., duplicate CREATE TABLE), which allowed the schema conflict to go undetected. Real SQLite correctly rejects invalid SQL.

3. **Deferred table existence checks**: The runner re-checks if `_migrations` table exists after each migration runs. This catches if a migration fails to create the required table, providing fast feedback.

4. **Convention over configuration**: Migrations must be:
   - Numbered files (e.g., `0001_init.sql`, `0002_users.sql`)
   - Forward-only (no down migrations)
   - The first migration must have "init" in its name
   - Never edited after deployment (enforced by convention in README)
