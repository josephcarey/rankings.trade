# Runbook — Deploy (app & cron workers)

> rankings.trade ships as **two** Cloudflare Workers that share one D1 database: the SvelteKit
> **app** worker and the 15-minute **cron** scrape worker. Deploying one does **not** deploy
> the other. See the [README — Two-worker split](../../README.md#two-worker-split-app-vs-cron)
> for the architectural rationale.

## 1. What deploys where

| Worker | Config | Entry | Deploy command | Cloudflare name |
|---|---|---|---|---|
| App | `wrangler.toml` | `.svelte-kit/cloudflare/_worker.js` (generated by `bun run build`) | `bun run deploy` (`wrangler deploy`) | `rankings-trade` |
| Cron | `wrangler.cron.toml` | `worker/cron.ts` | `bun run deploy:cron` (`wrangler deploy -c wrangler.cron.toml`) | `rankings-trade-cron` |

Both bind the **same** D1 database (`DB` → `rankings-trade-prod`). The cron worker only runs a
`scheduled` handler on the `*/15 * * * *` trigger; it serves no HTTP traffic.

## 2. Preconditions

- [ ] Change is merged to `main` and CI is green (`bun install && bun run ci`).
- [ ] Any **new D1 migrations** in the change are accounted for (see §4 — migrations are applied
      separately from the worker deploy).
- [ ] Cloudflare credentials are available to the deploy environment: `CLOUDFLARE_API_TOKEN`
      and `CLOUDFLARE_ACCOUNT_ID` (CI deploy reads these from secrets; for a manual deploy,
      export them or be logged in via `wrangler login`).
- [ ] Worker runtime secrets are set in Cloudflare (these are **not** in `wrangler.toml`):
      `CLERK_SECRET_KEY` (app). Set/rotate with `wrangler secret put <NAME>` against the
      relevant config (`-c wrangler.cron.toml` for the cron worker if it needs one).
- [ ] `PUBLIC_CLERK_PUBLISHABLE_KEY` is present in `wrangler.toml` `[vars]` (it must be a Worker
      **runtime** var, not only build-time — see the README Clerk note).

## 3. Deploy

The normal path is the CI deploy job on merge to `main` (`wrangler deploy` via `bun run deploy`).
To deploy manually, or to deploy the cron worker (which CI may not cover), run from a clean,
up-to-date `main`:

```sh
bun install        # ensure deps + svelte-kit sync
bun run ci         # confirm green before shipping
bun run build      # generate the Cloudflare worker bundle (app)

# App worker:
bun run deploy

# Cron worker (separate — does NOT happen as part of `bun run deploy`):
bun run deploy:cron
```

> **Deploy the cron worker whenever scrape logic, its bindings, or the schedule change.** It is
> easy to forget because `bun run deploy` says nothing about it.

## 4. Migrations

Worker deploys do **not** run database migrations. If the change includes new files under
`migrations/`, apply them against production D1 **after** verifying them locally:

```sh
# Locally first, against the local DB binding (wrangler dev running in another terminal):
bun run db:migrate

# Production: apply each new migration to the prod D1 via wrangler, e.g.
wrangler d1 execute rankings-trade-prod --remote --file migrations/0018_example.sql
```

Apply migrations in numeric order and confirm `_migrations` records them. Never edit a shipped
migration (see [`migrations/README.md`](../../migrations/README.md)).

## 5. Verify (after deploy)

App worker:

- [ ] `GET /api/health` returns `{ "status": "ok" }` (e.g. `curl https://rankings.trade/api/health`).
- [ ] The home page and a public profile/leaderboard route render (no SSR 500 — a missing Clerk
      publishable key shows as "Publishable key is missing").
- [ ] Cloudflare dashboard → the `rankings-trade` worker shows the new deployment/version.

Cron worker:

- [ ] Cloudflare dashboard → `rankings-trade-cron` shows the new version and an **active**
      `*/15 * * * *` trigger.
- [ ] Within ~15 minutes, a scheduled invocation succeeds (Workers logs / "Cron Triggers" tab),
      and fresh `snapshots` rows appear for the current round.

## 6. Rollback

- **Worker:** use Cloudflare's **Deployments → Rollback** to revert either worker to its previous
  version (do this per-worker — app and cron roll back independently). Alternatively, check out
  the last-good commit and re-run the relevant `bun run deploy` / `bun run deploy:cron`.
- **Schedule incident:** if a bad cron deploy is scraping incorrectly, roll back the cron worker
  (above) or temporarily disable its trigger in the Cloudflare dashboard while you fix forward.
- **Migrations:** migrations are forward-only and **cannot** be rolled back by redeploying an
  older worker. Restore from a D1 backup/export if a migration must be undone, then redeploy the
  matching worker version. Take an export before applying risky migrations.

## 7. Notes

- App and cron versions can legitimately differ; keep both current when a change spans shared
  code (e.g. `src/lib/scrape/scheduled.ts`, the D1 schema).
- Secrets live only in Cloudflare and CI — never in `wrangler.toml`, `.dev.vars` (git-ignored),
  or commits.
